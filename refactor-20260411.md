# 2026-04-11 重构方案

这份文档用于承接 2026-04-11 之后的重构工作。

目标不是回溯 `v0` 已完成阶段，而是为后续 session 提供一份稳定、可执行、可持续更新的 refactor backlog。

后续如无额外说明，优先按本文档执行。

---

## 原始建议

用户提出的原始优化方向如下：

1. 前端拆一个独立项目出来，前后端之间明确用 API
2. 前端界面大幅简化，不要复杂动效背景，简洁直接，并支持黑暗模式
3. 把 `as` 和 `unknown` 改成明确领域对象，补充 `zod` 校验；类型可考虑单独定义并前后端共用
4. 后端用 `Effect` 重写

---

## 当前判断

总体判断：

- 第 1 项：赞成
- 第 2 项：赞成
- 第 3 项：强烈赞成，且优先级最高
- 第 4 项：不建议立即全量重写，只建议后续按模块增量引入

核心原因：

- 当前前端仍内嵌在 `apps/api` 中，页面渲染和路由、数据查询强耦合，继续演进会拖慢 API 契约收敛和前端迭代速度
- 当前 UI 已支持明暗主题，但样式层明显偏重，不符合“简洁直接”的新目标
- 当前真正的结构性问题不是“语法层面有几个 `as`”，而是评测结果、提交详情等关键 JSON 结构从 DB 到 API 到 UI 没有形成稳定领域类型
- 当前后端代码规模仍可控，立刻全量迁移到 `Effect` 会把“类型收敛”“前后端解耦”“服务边界梳理”三件更重要的事情全部绑在一次高风险改造里

---

## 建议顺序

建议执行顺序如下：

1. 先建立共享契约层，收紧领域类型与 `zod` schema
2. 再拆独立前端项目，并把界面同步简化
3. 然后重整 API 分层，让后端围绕契约层稳定输出
4. 最后只在合适模块试点引入 `Effect`，而不是一次性全仓重写

这个顺序的原则是：

- 先钉死契约，再拆边界
- 先减少不确定性，再替换框架风格
- 先解决真实痛点，再处理“技术品味型重构”

---

## 各项建议的具体判断

### 1. 独立前端

结论：应做。

原因：

- 现在的页面渲染直接放在 `apps/api/src/ui.ts`
- API 服务同时承担 HTML 页面、静态资源、路由和数据装配职责
- 后续如果要做更清晰的人类端体验、组件化界面、前端测试和独立部署，这个边界必须先拆

建议落点：

- 新建 `apps/web`
- `apps/api` 只提供 HTTP API 与静态资源最小职责
- 人类前端只通过 API 获取数据
- 保留 server 端 admin / worker / DB 架构，不急着拆更多服务

### 2. 前端简化

结论：应与前端拆分同步推进。

原因：

- 现在样式明显偏重，装饰性背景、阴影、面板层级和视觉噪声较多
- 本项目的真实重点是题目、提交、评测结果、排行榜，不是展示型视觉设计
- 简化后更利于后续维护、信息阅读和移动端适配

建议落点：

- 保留黑暗模式与亮色模式
- 保留必要的信息层级和少量状态色
- 去掉复杂背景、过重阴影、装饰性排版
- 页面优先强调信息密度、可扫描性和可维护性

### 3. 领域类型 + Zod

结论：这是第一优先级。

原因：

- 当前 `unknown` / `as` 主要集中在评测结果、summary、UI 展示等关键业务路径
- 这些结构一旦不稳定，前端拆分出来后仍然会继续痛苦
- 仓库已经在 `packages/shared` 里使用 `zod` 管理环境变量和题目契约，继续扩展到 API / DTO / evaluation 结构是自然延伸，不是推倒重来

建议落点：

- 新建单独的共享契约包，例如 `packages/contracts`
- 统一放：
  - API request/response schema
  - evaluation 领域对象
  - leaderboard / submission / discussion DTO
  - 可能需要前后端共用的枚举、ID 字段和 payload schema
- DB 层不再向上游暴露“未建模 JSON”
- API 层不再把 `unknown` 直接原样透传给前端
- 前端只消费经过 schema 约束的领域对象

### 4. 后端 Effect 化

结论：暂不全量执行，只做后续候选。

原因：

- 当前最大问题不在“异步组合能力不够”，而在契约不闭合、边界不清楚、模块职责偏厚
- 全量迁移 `Effect` 的改造面会非常大，且会横跨 API、DB、worker、配置、日志、错误处理
- 如果在类型与边界还没收稳前就先做 `Effect`，迁移中会不断返工

建议策略：

- 先完成契约层、独立前端、API 分层
- 再选择 1 个高收益模块试点 `Effect`
- 试点优先考虑：
  - worker job 执行链路
  - 带事务的数据库服务
  - 需要明确错误类型的评测回写流程

明确不建议：

- 当前阶段直接把整个仓库改成 `Effect`
- 一边拆前端一边做全量 `Effect`
- 在没有明确收益验证前，把所有简单 CRUD 也一起迁走

## 后续任务清单

说明：

- 本任务清单是后续 refactor session 的执行基线
- 除非中途发现阻塞性新信息，否则按顺序推进
- 单个阶段完成后再进入下一阶段，避免并行大改导致回归面过大

### Phase R1: 契约层与领域类型收敛

当前状态：已完成（2026-04-11，本轮已验收并提交）。

#### 阶段目标

先把前后端共享的核心业务结构收成稳定契约，减少 `unknown` 与断言扩散。

#### 具体任务

- [x] 新建 `packages/contracts`
- [x] 迁移或抽取评测相关 schema：
  - [x] shown case result
  - [x] hidden summary
  - [x] official summary
  - [x] evaluation record
- [x] 定义 submission 相关 API DTO schema
- [x] 定义 problem / leaderboard / discussion API DTO schema
- [x] 为 API request body 增加统一 `zod` 校验入口
- [x] 让 DB 层返回明确领域对象，而不是 `unknown`
- [x] 替换 UI / API 中对评测结果的 `Record<string, unknown>` 与局部 `as`
- [x] 补充 contracts 层单测

#### 完成门禁

- 评测结果的核心类型有统一 schema 和类型导出
- 现有功能行为不回退

### Phase R2: API 分层整理

#### 阶段目标

在独立前端拆出之前，先把 API 服务内部职责切清楚。

#### 具体任务

- [ ] 将 `apps/api` 中路由、鉴权、数据装配、页面逻辑进一步解耦
- [ ] 明确 route handler 只负责：
  - [ ] 读取输入
  - [ ] 调用 service
  - [ ] 返回 contracts DTO
- [ ] 视需要抽出 `services/` 或 `presenters/`
- [ ] 减少 `app.ts` 的体积与横向职责
- [ ] 为关键接口补充基于 contracts 的响应测试

#### 完成门禁

- `apps/api/src/app.ts` 不再同时承担过多业务装配细节
- API 输出结构由 contracts 驱动

### Phase R3: 独立前端项目

## 任务项变更点

这里的“变更点”指相对于当前仓库已有 `tasks.md` 的新增重构工作，而不是覆盖已有 `v0` 历史阶段。

### 任务组织方式变更

- 现有 `tasks.md` 保持为 `v0 prototype` 的历史开发拆分，不直接改写为 refactor 文档
- 新增一条独立重构主线，使用 `Phase R1` 到 `Phase R5`
- 后续 session 默认优先参考本文档，不再把重构任务硬塞进旧的 `Phase 0` 到 `Phase 8`

### 任务重点变更

- 从“先做功能闭环”转为“先收紧契约与边界，再重构体验层”
- 将“清理 `as` / `unknown`”升级为第一优先级，而不是零散代码清洁项
- 将“前端拆独立项目”放在契约层之后执行，避免先拆再返工
- 将“前端视觉简化”明确为独立阶段目标，而不是前端拆分时顺手处理
- 将“Effect 重写”降级为试点评估项，而不是立即立项的大规模改造

### 验证方式变更

- 每个重构阶段仍需保留：
  - [ ] `npm run lint`
  - [ ] `npm run typecheck`
  - [ ] `npm run build`
  - [ ] `npm test`
- 如阶段引入新前端工具链，需要补该工具链对应验证命令
- 如阶段涉及人工界面验收，需要在阶段记录中补截图或 showboat 证明
